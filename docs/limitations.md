# Known Limitations

As a Proof of Concept, this application is designed to demonstrate core RAG capabilities in an IT helpdesk context. The following limitations exist by design:

* **No Production Database Access:** The AI does not have read/write access to live active directories, ticketing software, or employee databases. It cannot execute actions for the user, though it can guide them through certain procedures (e.g., password resets).
* **Email-based Escalation Technique:** This system cannot send tickets to a hypothetical human helpdesk and rather relies on the user sending an email via a pre-formatted `mailto:` link. Production-stage architecture could replace this with a direct REST API call to MDA's ticketing platform (e.g., ServiceNow or Jira) for background ticket generation.
* **Ephemeral Session Memory:** The application utilizes AWS Bedrock's native session handling to maintain conversational context during an active chat. However, for security purposes in this PoC, this memory is *ephemeral* and wiped upon refreshing the page or clicking "Clear Chat History." In a production release, conversational state could be logged to Amazon DynamoDB for persistent cross-session memory.
* **Manual Data Syncing:** The S3-to-OpenSearch ingestion pipeline currently requires a manual sync trigger via the AWS console when new documents are added.
* **Sensitive Keys:** The Knowledge Base ID and Administrator Password is solely stored by the .env file which is not up to security standards for production-stage implementations. Production-stage deployments could migrate these secrets to AWS Secrets Manager and utilize IAM Role-Based Access Control (RBAC) via Active Directory (SSO).
* **False Positives in Audit Log:** The 'Mailto Triggered' field of the audit log is highly susceptible to false positives as the system checks if the `mailto:` link was generated by the AI and not whether the user clicked the link and/or truly sent the email. A production-stage implementation could utilize an AWS API Gateway redirect to track physical clicks rather than frontend generation.
* **Missing Information in Audit Log:** There are some missing fields in the audit log which would otherwise be useful in a production-stage implementation including, but not limited to, who made the inquiry (user).
